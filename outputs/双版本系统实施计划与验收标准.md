# AI 辅助环评前期研判系统双版本实施计划与验收标准

制定时间：2026-07-19  
实施对象：单机版、私有化服务器版  
共同业务内核：HTML 原型前端 + FastAPI/LangGraph + Hermes Agent 0.18.0

## 1. 实施原则

1. 不重复开发 Agent 底层能力。文档读取策略、OCR/视觉、Web Search、浏览器、终端、代码执行、子 Agent、Runs/SSE/stop 和审批继续使用 Hermes 原生能力。
2. LangGraph 只管理环评业务节点、暂停恢复、checkpoint 和节点顺序；不替代 Hermes 的节点内自主执行。
3. Docker/Compose 负责交付和隔离，Caddy 负责服务器入口的 TLS、压缩和基础认证，不在业务后端手写反向代理或密码系统。
4. 两个版本共享业务代码和镜像定义，不建立长期分叉代码；差异只放在 Compose、环境变量、运行脚本和运维文档中。
5. 服务器版定位为“私有化单租户/团队共享部署”，不宣称已经支持公网多租户 SaaS。多租户需要独立身份系统、任务归属和数据库升级，另立项目。

## 2. 目标架构

### 2.1 共同业务层

```text
浏览器
  -> FastAPI API 与静态前端
  -> LangGraph 业务状态机
  -> Hermes Runs API
  -> Hermes 原生工具与模型
```

后端仅负责上传、任务状态、提示词变量、Hermes 事件转发、结果归档和知识依据治理；不实现 PDF 阅读器、OCR 引擎、视觉模型客户端、搜索引擎或通用 Agent 循环。

### 2.2 单机版

```text
浏览器 localhost
  -> backend-desktop 容器
  -> hermes-desktop 容器
       -> Hermes local terminal（只在该容器内执行）
       -> PDF/OCR/视觉/Web/子 Agent
```

- 全部服务由 Docker Compose 管理，不依赖宿主机 Hermes 或 tmux。
- Web 和 Hermes 调试端口只绑定 `127.0.0.1`。
- Hermes Controller 与 terminal 共用容器文件系统，直接解决视觉工具读取 PDF 渲染图片的路径问题。
- 适合单人可信环境；不作为多人服务器安全边界。

### 2.3 私有化服务器版

```text
用户
  -> Caddy（TLS + Basic Auth）
  -> backend-server 容器
  -> hermes-server Controller 容器
       -> Hermes 原生 Docker terminal
            -> 每个 task_id 独立工具容器
```

- Hermes Controller 使用官方容器运行和 s6 监督机制。
- 通过 Docker socket 使用 Hermes 原生 Docker terminal，不建设自研 Worker。
- 工具容器使用独立文档工具镜像、cap-drop、no-new-privileges、资源限制、无模型密钥。
- 后端与 Hermes API 不直接发布到公网，只由 Caddy 提供入口。
- 服务器版使用单后端进程和本地持久化，适用于单租户低并发；不支持横向扩容。

## 3. 已发现隐患与处置

| 隐患 | 处置方案 | 复用能力 |
| --- | --- | --- |
| 默认 Compose 依赖宿主机 Hermes/tmux | 单机版和服务器版均容器化 Hermes Controller | Hermes 官方 Docker Gateway/s6 |
| 现有“全容器模式”使用 `terminal.backend: local` 且定位不清 | 仅单机版使用 local；服务器版使用 Hermes Docker terminal | Hermes terminal backend |
| 后端 8501 对所有网卡开放且无认证 | 单机版绑定 localhost；服务器版仅 Caddy 发布，启用 TLS/Basic Auth | Docker 端口绑定、Caddy |
| CORS 完全开放 | 改为环境变量控制，默认同源 | FastAPI CORS 中间件 |
| Docker 图片路径无法交给 Controller 视觉工具 | 增加受控视觉缓存，提示词同时提供沙箱写入路径和 Controller 可见路径 | Hermes `vision_analyze` + `docker_volumes` |
| 容器重启后运行中协程丢失 | 后端启动时自动将孤立任务安全恢复为 paused，并停止遗留 Hermes run | LangGraph checkpoint + Hermes run stop |
| 工具容器写出 root 文件 | 后端使用宿主 UID/GID；服务器 Docker terminal 开启 `docker_run_as_host_user` | Hermes 原生配置 |
| 工具镜像依赖后端镜像且夹带应用代码 | 改为独立、固定用途的文档工具镜像 | Docker 镜像分层 |
| 固定容器名、固定数据目录导致双版本冲突 | Compose project name、端口和 runtime 根目录完全分离 | Docker Compose |
| 镜像现场构建受网络影响 | 国内 apt/PyPI 默认源，提供预构建/导出镜像脚本 | Docker save/load |
| JSON/SQLite 不支持多进程和多租户 | 服务器版明确单进程单租户；多租户升级不混入本次范围 | 当前 LangGraph SQLite saver |
| 共享项目资料目录不满足租户隔离 | 服务器版限定单租户；不同租户必须独立部署实例和 runtime 目录 | Compose project 隔离 |

## 5. 阶段验收标准

### DUAL-01

- 计划中明确哪些能力来自 Hermes，哪些属于业务系统。
- 单机版、服务器版和多租户 SaaS 边界无混淆。
- 每项既有隐患有对应处置或明确非目标。

### DUAL-02

- 后端编译和前端语法检查通过。
- CORS 默认不再为 `*`，可通过环境变量配置。
- 后端重启后孤立 `running` 任务自动变为 `paused`，不自动跳过失败节点。
- Agent 提示词包含视觉缓存的沙箱路径和 Controller 路径。
- 文档工具镜像不包含业务源码和模型密钥。

### DUAL-03 单机版

- 只安装 Docker Desktop/Engine 即可启动，不要求宿主机安装 Hermes、Python 或 tmux。
- Web 与 Hermes 端口仅监听 `127.0.0.1`。
- Hermes API 健康，后端健康，前端同源请求正常。
- Agent 能读取 PDF、执行 Python/OCR，并能让 `vision_analyze` 读取同一文件路径。
- 数据、Hermes 状态、输出和日志重启后保留。

### DUAL-04 服务器版

- 后端和 Hermes 仅在 Compose 内部网络可见，主机不发布 8501/8642。
- 未认证访问返回 401；认证后可打开系统和调用 API。
- Hermes 使用 `terminal.backend: docker`，工具 run 产生 Hermes 标记的独立容器。
- 工具容器无模型 Key、无 Docker socket，项目输入只读、输出/视觉缓存可写。
- 任务暂停、容器重启和后端恢复路径可观测。

### DUAL-05

- 默认密钥或空模型配置会在启动前失败，不带病启动。
- 双版本可用不同 Compose project name、端口和 runtime 目录同时存在。
- 新建文件不再默认成为不可读的 root-only 成果。
- 备份不默认打包明文模型 Key；密钥另行提示备份。
- 可生成离线镜像包清单和 Docker image tar。

### DUAL-06

- `docker compose config`、Shell 语法、Python 编译、前端 JS 检查全部通过。
- 两个版本分别完成容器健康验收。
- 至少一次真实 Hermes run 验证模型、terminal、PDF/OCR/视觉路径和结果回收。
- 服务验收过程中没有未处理的 `approval.request`、401、路径不可见或后台继续越过失败节点。

## 6. 发布判定

- 单机版通过 DUAL-01 至 DUAL-06 后，可交付给单个非技术用户试用。
- 服务器版通过 DUAL-01 至 DUAL-06 后，可用于受控内网或 VPN 后的单租户团队。
- 公网多租户、按用户隔离任务、横向扩容和高可用不在本次发布声明中；未完成这些能力前不得宣传为 SaaS 版本。
