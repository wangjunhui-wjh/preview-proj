## 实际运行路径

当前浏览器访问的服务为历史兼容部署：

```text
eia-ai-backend:latest
  -> http://host.docker.internal:8642
  -> 宿主 Hermes Gateway
  -> custom provider / grok-4.5
```

这不是当前新交付的 `deploy/desktop/` 单机版。Gateway `/health` 正常；使用同一内部 Gateway Key 的最小 `/v1/chat/completions` 请求在诊断时返回 HTTP 200 且内容为 `OK`。因此模型服务并非全局宕机，502 是该次长工具上下文请求被上游拒绝或其代理转发失败。

失败时 Hermes 记录的请求规模约为 20 条消息、约 22,305 tokens。不能从上游返回体进一步区分限流、代理故障、模型临时异常或该轮消息格式兼容性问题，因为上游仅返回通用 `upstream_error`。

## 同时发现的视觉路径问题

`PREP-INGEST` 将 PDF 页面渲染到 Docker terminal 私有路径：

```text
/workspace/prep_ingest/pages/flow-1.png
/workspace/prep_ingest/pages/flow-2.png
```

随后 `vision_analyze` 在宿主 Hermes Controller 中执行，无法访问工具容器的 `/workspace`，因此两次返回 `Invalid image source`。这不是 502 的直接返回源，但会向模型对话加入失败工具结果并降低本轮稳定性。

## 建议处理

1. 不要把失败任务自动向后推进；保留在 `PREP-INGEST` 重跑边界。
2. 使用新单机版 `deploy/desktop/` 时，通过其 `.env` 配置实际要使用的模型；它不依赖当前宿主 `grok-4.5` Gateway。
3. 使用服务器版时，保持 Hermes 原生 Docker terminal 的 `/eia/vision-cache/<task_id>` 视觉缓存交接；不要让 Agent 对工具容器私有 `/workspace` 图片调用 Controller vision。
4. 若必须继续使用当前历史栈，先等待或切换上游模型后再从 PREP-INGEST 重跑；最小请求成功不代表长工具上下文可稳定通过该上游代理。

本次诊断未改变运行中的模型、Hermes 配置、任务状态或容器。
