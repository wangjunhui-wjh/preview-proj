# 环评前期研判 AI 助手实施方案

## 1. 技术路线

当前路线暂定为：

```text
前端原型：环评前期研判AI助手.html
后端服务：FastAPI
流程状态机：LangGraph
节点执行 Agent：Hermes Agent
任务状态：LangGraph checkpoint + data/tasks/
上传文件与工作区：data/uploads/ + data/workspaces/
运行日志：logs/
节点产物：outputs/
提示词资产：prompts/
```

核心原则：

- 不再在业务系统里手写通用 PDF 解析、OCR、图片识别、Web Search 工具。
- 每个 HB 节点内部的文档读取、网页搜索、图片/OCR、工具调用交给 Hermes Agent。
- LangGraph 只负责外层流程控制、状态、暂停、恢复、失败停止和节点输入输出传递。
- 前端 HTML 负责展示和交互，不再直接持有大模型 Key，不再直接拼接完整后台执行逻辑。

## 1.1 双版本交付边界

业务代码不再以“后端容器连接宿主机 Hermes”作为部署方式。交付统一由 `deploy/` 管理：

- `deploy/desktop/` 是完整容器化单机版，面向一名可信用户，Web 与 Hermes 端口仅绑定本机回环地址。
- `deploy/server/` 是完整容器化私有化服务器版，面向单一组织的内网/VPN/受控服务器；仅 Caddy 对外发布，backend 与 Hermes 不发布主机端口。
- 两个版本都复用 Hermes 原生 Agent 与 terminal 能力，使用不同的 Compose 项目名、运行目录和 `.env`；不依赖也不修改宿主机 Hermes。
- 服务器版是单租户单后端进程，不宣称已具备多用户隔离、横向扩容或高可用。详细计划、风险和验收见 `outputs/双版本系统实施计划与验收标准.md`。

## 2. 现有原型定位

`环评前期研判AI助手.html` 是目标产品原型，保留为交互基准。

原型中值得保留的设计：

- 左侧 HB-PT 步骤导航。
- 资料准备、知识库、联网检索、API 配置、模块结果、综合报告、交叉核查等页面结构。
- 每个模块独立运行、清除、复制、导出。
- 模块状态显示和结果历史查看。
- 自动联网检索、知识依据注入、政策有效性检查等产品意图。

需要调整的地方：

- 前端不再直接调用模型 API，改为调用后端任务 API。
- `localStorage` 只作为页面草稿缓存，不作为任务真实状态。
- API 配置页面不再暴露大模型密钥，模型/Hermes 配置走后端环境变量。
- Web Search 关键词不由前端固定，节点 agent 根据任务和资料自行组织搜索。
- 后台思维链不展示；页面展示工具调用轨迹、读取文件、访问 URL、阶段性结论和错误信息。
- 文件上传必须进入后端任务工作区，由 Hermes 节点读取，不在浏览器内完成核心解析。

## 4. 数据存储与知识库设计

系统存储分四层，避免把“项目资料”“政策依据”“搜索候选”和“运行记录”混在一起。

```text
项目库
  - 用户上传的项目 PDF/DOCX/图片/表格
  - 粘贴或编辑的项目简介
  - 任务工作区、中间结果、最终报告

政策知识库
  - 已确认有效的政策、法规、名录、技术规范、规划环评、准入清单
  - 保留版本、文号、发布机关、发布日期、有效性和废止/替代关系

候选依据库
  - Web Search 找到但尚未确认的网页、PDF、附件、转载线索
  - 只能作为候选依据或检索线索，不能直接当正式政策库使用

运行记录库
  - 每个节点的 Hermes run_id、事件流、工具调用、引用依据和输出产物
```

推荐 MVP 使用文件系统 + SQLite，正式部署升级为 PostgreSQL + 对象存储 + 向量/全文索引。

MVP 目录：

```text
data/
  uploads/                         用户上传的项目资料
  workspaces/{task_id}/             Hermes 节点工作区
  tasks/{task_id}/                  任务状态快照，供前端/API 恢复展示
  langgraph_checkpoints.sqlite      LangGraph 外层流程 checkpoint
  knowledge/
    policy_files/                   政策原文 PDF/DOCX/HTML
    web_snapshots/                  网页快照、响应头、访问时间
    extracted_text/                 OCR/解析后的纯文本
    indexes/                        FTS/向量索引文件
    candidates/                     Web Search 候选依据暂存
```

正式部署建议：

```text
PostgreSQL
  tasks
  task_files
  task_nodes
  node_events
  evidence_refs
  web_search_records
  knowledge_documents
  knowledge_versions
  knowledge_chunks
  knowledge_ingestion_jobs
  policy_validation_records

对象存储或文件系统
  原始 PDF/DOCX/图片
  网页 HTML 快照
  OCR 图片和中间文件
  节点输出 JSON/Markdown

检索层
  PostgreSQL full-text / SQLite FTS
  pgvector / Qdrant / Milvus
```

核心实体：

```json
{
  "knowledge_document": {
    "id": "uuid",
    "title": "建设项目环境影响评价分类管理名录（2021年版）",
    "source_type": "official_url|uploaded_file|web_candidate",
    "source_url": "https://...",
    "source_domain": "mee.gov.cn",
    "issuer": "生态环境部",
    "doc_no": "...",
    "published_at": "...",
    "retrieved_at": "...",
    "file_hash": "sha256:...",
    "status": "candidate|verified_candidate|verified|rejected|deprecated",
    "validity": "effective|superseded|expired|unknown",
    "supersedes": [],
    "superseded_by": null
  }
}
```

状态规则：

- `candidate`：搜索发现或用户上传，但未校验。
- `verified_candidate`：来源可靠、原文已保存、元数据基本完整，但等待人工确认。
- `verified`：已确认可作为正式政策库依据。
- `rejected`：来源不可靠、重复、内容不相关或无法追溯。
- `deprecated`：被新文件替代、废止或过期。

## 5. 政策库更新与候选入库流程

研判过程中允许 Hermes 通过 Web Search 发现新政策文件，但不能直接把搜索摘要写入正式政策库。必须经过候选入库、校验和确认。

流程：

```text
节点发现依据不足
  -> Hermes 自主组织 Web Search 查询词
  -> 返回真实 URL、标题、摘要、访问时间
  -> 后端创建 web_search_record
  -> 后端下载原文或保存网页快照
  -> 抽取文本/OCR/解析 PDF
  -> 提取标题、发布机关、文号、发布日期、施行日期、有效性线索
  -> 计算文件 hash 和去重
  -> 切块并建立全文/向量索引
  -> 写入 knowledge_documents(status=candidate)
  -> 来源校验通过后提升为 verified_candidate
  -> 人工确认或规则确认后进入 verified
```

自动校验规则：

```text
官方域名 + 可保存原文 + 元数据完整
  -> verified_candidate

政府/生态环境部门子站、法规库、标准库
  -> verified_candidate，但需要记录来源层级

非官方转载、公众号、文库、论坛、商业网站
  -> candidate/search_clue，只能作为线索，不进入正式政策库

用户人工上传的政策文件
  -> candidate，等待人工确认来源和有效性
```

当前节点可以引用 `candidate` 或 `verified_candidate`，但报告中必须标注：

- 来源 URL 或文件名。
- 检索时间。
- 是否已进入正式政策库。
- 是否需要人工核实。

正式报告默认优先引用 `verified`；没有 `verified` 时，才允许引用 `verified_candidate`，并提示“建议人工核实”。普通 `candidate` 不应作为强结论依据。

入库去重：

- 原文件 hash 相同：直接复用已有文档。
- URL 不同但标题、文号、发布机关、正文 hash 相同：合并为同一文档的多个来源。
- 同标题不同发布日期或文号：作为不同版本，建立版本关系。

检索顺序：

```text
先查本地 verified 政策库
  -> 不足时查 verified_candidate
  -> 仍不足时由 Hermes Web Search
  -> 搜到新文件后进入 candidate/verified_candidate
  -> 节点输出记录 evidence_refs
```

知识库更新不能覆盖任务历史。已经完成的任务应继续引用当时的文件版本和 hash；政策库后续更新只影响新任务或用户主动重跑的节点。

## 6. LangGraph 状态设计

核心 state：

```python
class EiaTaskState(TypedDict):
    task_id: str
    status: Literal["created", "running", "paused", "failed", "completed"]
    current_node: str | None
    next_node: str | None
    project_text: str
    project_files: list[FileRef]
    evidence_files: list[FileRef]
    knowledge_doc_ids: list[str]
    candidate_doc_ids: list[str]
    selected_modules: list[str]
    module_results: dict[str, NodeResult]
    evidence_refs: list[EvidenceRef]
    events: list[TaskEvent]
    error: str | None
    created_at: str
    updated_at: str
```

固定主链路：

```text
prepare
  -> PREP-INGEST
  -> HB-PT-000
  -> HB-PT-001
  -> route_core_modules
  -> HB-PT-002
  -> HB-PT-003
  -> HB-PT-005
  -> HB-PT-007
  -> HB-PT-010
  -> HB-PT-011
  -> complete
```

分支规则：

- `PREP-INGEST` 读取用户粘贴文本、PDF/DOCX/扫描件/图片，生成带来源索引的项目档案。
- `HB-PT-000` 基于项目档案输出资料完整性审查和建议启动模块。
- `route_core_modules` 根据 `selected_modules` 决定后续是否跳过某些节点。
- 任一节点失败：`status=failed`，不进入后续节点。
- 用户暂停：运行中的任务先保存 `pause_requested=true`，再请求停止当前 Hermes run；停止后 `status=paused`，`next_node` 保持当前节点，便于后续重跑或恢复。
- 用户恢复：从 `next_node` 继续执行。
- 用户重跑某节点：清理该节点及其后续依赖结果，再从该节点继续。
- 任务引用的知识库版本、文件 hash 和 URL 快照必须固化到任务状态，避免后续政策库更新改变历史结论。

## 7. Hermes 节点执行方式

每个流程节点是一个 Hermes 任务，不是一次普通 LLM 请求。`PREP-INGEST` 负责项目资料读取与项目档案构建，后续 HB 节点负责专项研判。

节点执行输入：

```text
系统提示词：prompts/system_prompt.txt
节点提示词：prompts/prep_ingest_project_dossier.txt 或 prompts/hb_pt_xxx_*.txt
项目材料：PREP-INGEST 使用 project_text + project_files；后续 HB 节点优先使用 PREP-INGEST 输出的项目档案
已完成模块：module_results
约束：
  - PREP-INGEST 自动读取需要的 PDF/DOCX/图片/表格，并建立来源索引
  - 后续 HB 节点不把浏览器解析文本当正式项目事实，优先读取项目档案
  - 必要时进行网页搜索
  - 优先检索本地 verified 政策库，再使用 Web Search
  - Web Search 找到的新政策先进入候选依据库
  - 只引用真实读取到的文件或真实访问过的 URL
  - 输出固定 JSON + Markdown
  - 依据不足必须标注资料不足
```

Hermes 工作区：

```text
data/workspaces/{task_id}/
  project_files/
  evidence_files/
  knowledge_refs/
  candidate_refs/
  prompts/
  previous_results/
  node_output/
```

每个节点调用 Hermes 前，后端把相关文件和提示词写入工作区，然后发送任务说明：

```text
请执行 HB-PT-002 行业类别、环评类别及审批路径判定。
你可以读取 workspace 内项目资料，可以使用文档读取、OCR、Web Search、网页抓取等工具。
必须返回：
1. node_result.json
2. node_result.md
3. evidence_refs.json
4. tool_trace.json
```

输出校验：

- `node_result.json` 必须能被 Pydantic schema 解析。
- `evidence_refs.json` 中每条依据必须有 `source_type=file|url|tool`。
- URL 依据必须有真实 URL。
- 文件依据必须有文件名，能提取页码则记录页码。
- 政策库依据必须记录 `knowledge_document_id`、版本号或文件 hash。
- 校验失败时允许 Hermes 自动修复一次；仍失败则节点失败并停止流程。

### 7.1 Hermes 服务化运行方式

Hermes 不只支持交互式 CLI。它的 API Server 是 `gateway` 的一个 platform，可以通过 `hermes gateway run` 前台运行，也可以通过 `hermes gateway install/start` 安装并启动为 systemd/launchd 后台服务。

推荐本系统采用：

```text
FastAPI/LangGraph 后端 -> Hermes API Server -> Hermes Agent -> 模型与工具后端
```

最小服务配置：

```bash
export API_SERVER_ENABLED=true
export API_SERVER_KEY=change-me
export API_SERVER_HOST=127.0.0.1
export API_SERVER_PORT=8642
hermes gateway run
```

可用接口包括：

- `POST /v1/chat/completions`
- `POST /v1/responses`
- `POST /v1/runs`
- `GET /v1/runs/{run_id}/events`
- `POST /v1/runs/{run_id}/stop`
- `GET /health`

实施时优先调用 `/v1/runs` 与 SSE 事件流，便于前端显示实时进度、工具事件和节点执行状态；暂停/停止映射到 `/v1/runs/{run_id}/stop`，LangGraph 再决定是否继续后续节点。

## 8. API 设计

前端只调用后端 API。

```text
POST   /api/tasks
       创建任务，上传项目文本和文件

GET    /api/tasks/{task_id}
       获取任务状态、当前节点、已完成结果

POST   /api/tasks/{task_id}/run
       从当前 next_node 开始连续执行

POST   /api/tasks/{task_id}/step
       只执行一个节点

POST   /api/tasks/{task_id}/pause
       请求暂停；运行中任务会停止当前 Hermes run 并停在当前节点

POST   /api/tasks/{task_id}/resume
       从 checkpoint 恢复

POST   /api/tasks/{task_id}/rerun/{node_id}
       重跑指定节点

GET    /api/tasks/{task_id}/events
       SSE 或 WebSocket 实时事件流

GET    /api/tasks/{task_id}/outputs/{file_name}
       下载报告或节点产物

GET    /api/knowledge/documents
       查询政策知识库和候选依据库

POST   /api/knowledge/ingest
       手动提交 URL 或上传政策文件进入候选入库流程

POST   /api/knowledge/documents/{doc_id}/verify
       人工确认候选依据，提升为正式政策库文档
```

事件流类型：

```text
task_created
node_start
hermes_call_start
tool_event
file_read
web_search
url_fetch
knowledge_search
knowledge_ingest
knowledge_verified
node_output_partial
node_complete
node_failed
task_paused
task_completed
```

## 9. 前端优化方案

第一阶段不重做 UI，只把原型接后端。

必须改：

- 把前端模型 API 配置改成“后端连接状态”。
- 上传资料后调用 `/api/tasks`，返回 `task_id`。
- “运行分析”改为 `/step` 或 `/run`。
- 进度和日志从 `/events` 实时读取。
- 每个模块结果从 `/api/tasks/{task_id}` 读取。
- 页面刷新后用 `task_id` 恢复任务，不依赖 localStorage 里的完整结果。

建议优化：

- 左侧节点增加四种状态：未开始、运行中、已完成、失败。
- 每个节点结果页增加四个 Tab：结论、依据、工具轨迹、原始产物。
- 上传资料页显示后端已接收文件、Hermes 工作区路径、文件大小和类型。
- 综合报告页从 `outputs/` 读取最终产物。
- 知识依据页面区分“用户上传依据”和“Hermes Web Search 依据”。
- 知识库页面增加“正式政策库”“候选依据”“本任务引用依据”三个视图。
- 候选依据显示来源 URL、抓取时间、发布机关、文号、hash、状态和人工确认入口。

不建议做：

- 不展示模型隐藏思维链。
- 不在浏览器内执行 PDF 解析/OCR。
- 不让前端限定 Web Search 查询词。
- 不把大模型 Key 存在 localStorage。
- 不让前端把非官方转载一键确认为正式政策依据，至少需要显示风险和确认记录。

## 10.1 Hermes 自定义 API 接入确认

Hermes 支持自定义 OpenAI-compatible 模型 API。官方文档说明：只要服务实现 `/v1/chat/completions`，Hermes 就可以通过 custom provider 指向该服务，适用于本地模型、GPU 推理服务、多供应商路由或第三方 API。

推荐配置方式：

```bash
hermes model
# Select "Custom endpoint (self-hosted / VLLM / etc.)"
# Enter: API base URL, API key, Model name
```

或手动写入 `~/.hermes/config.yaml`：

```yaml
model:
  provider: custom
  default: gpt-5.5
  base_url: https://api.example.com/v1
  api_key: ""
```

注意：

- Hermes 文档建议 custom endpoint 使用 `hermes model` 或 `model.base_url` 配置。
- `OPENAI_BASE_URL` 仍可被 Hermes 识别，但主要用于 `openai-api` provider；对 custom endpoint，不作为首选配置方式。
- 自定义 OpenAI-compatible endpoint 的 `base_url` 必须指向真实 `/v1` API 根路径；如果漏掉 `/v1`，可能出现非流式返回形态异常或流式空响应。
- Hermes 自身也可以开启 API Server，向本系统暴露 OpenAI-compatible `/v1/chat/completions`；本系统后端调用 Hermes，而不是直接调用底层模型。
- 自定义模型 API 只解决“模型供应商”问题；网页搜索、浏览器、OCR/视觉等工具能力仍需要在 Hermes 中配置对应 tool backend、技能或 MCP。

## 11. 验收标准

最小可用版本：

- 可以从 HTML 页面创建任务。
- 可以上传 PDF/DOCX/图片/文本文件。
- 可以运行 `PREP-INGEST` 一个节点生成项目档案。
- `PREP-INGEST` 由 Hermes 自动读取文件、OCR/视觉识别并输出结构化项目档案；`HB-PT-000` 基于项目档案做完整性审查。
- 页面实时显示节点开始、Hermes 调用、文件读取、网页搜索、节点完成。
- 节点失败后后续节点不继续。
- 刷新页面后可恢复任务状态。
- `logs/` 有事件日志。
- `outputs/` 有节点 Markdown/JSON 产物。
- Web Search 发现的新政策文件能进入候选依据库，保存 URL/快照/hash/抽取文本。

完整版本：

- 支持 PREP-INGEST + HB-PT-000/001/002/003/005/007/010/011 全流程。
- 支持暂停、恢复、单步执行、连续执行、节点重跑。
- 支持 Hermes Web Search 依据回填。
- 支持政策知识库检索、候选入库、人工确认、版本追踪。
- 支持最终报告导出。
- 所有依据可追溯到文件或 URL。
- 任务重启后可从 checkpoint 继续。

## 12. 风险和处理

| 风险 | 处理 |
| --- | --- |
| Hermes 文档读取效果不稳定 | 先做 HB-PT-000 POC；必要时给 Hermes 增加 Docling/LlamaParse/Unstructured 作为工具或 MCP |
| Hermes Web Search 结果不可控 | 节点提示词强制记录 URL、时间、摘要和置信度；无依据则输出资料不足 |
| 非官方转载污染政策库 | Web Search 结果先进入 candidate；只有官方来源或人工确认后才能进入 verified |
| 政策文件版本混乱 | 保存文号、发布日期、施行日期、hash、superseded_by/supersedes 关系 |
| 政策库更新影响历史任务 | 任务引用固定到知识库版本和文件 hash；历史任务不随知识库更新自动改变 |
| 节点输出不结构化 | Pydantic 校验失败后自动要求 Hermes 修复一次 |
| 长任务中断 | LangGraph checkpoint + data/tasks 快照恢复 |
| 前端刷新丢状态 | `task_id` 恢复，localStorage 只保存最近 task_id 和草稿 |
| 权限风险 | 每个任务独立 workspace；Hermes 只允许访问该 workspace |
| 思维链展示风险 | 只展示工具轨迹和中间结论，不展示隐藏推理链 |
