【辅助提示词 - FILE-VALIDATION 上传资料有效性与可用性验证】

请读取用户上传的项目资料文件和项目原始文本，验证这些资料是否可读、是否与本项目相关、是否可以作为后续环评前期研判依据。

【输入变量】
项目原始文本：
{project_text}

项目档案或已完成模块：
{module_outputs}

知识库依据片段：
{evidence_context}

【执行规则】
1. 必须逐个读取上传文件；PDF 应先尝试文本层抽取，再对扫描页、图片页或图件使用 OCR/视觉识别，不要把整份 PDF 简单当作图片处理。
2. 判断文件类型和用途：项目事实资料、政策依据、规划/园区依据、生态环境分区管控依据、同类项目案例、图片/图纸/扫描件、无关资料、无法读取资料。
3. 对项目事实资料，应判断是否与本项目名称、建设单位、建设地点、产品、工艺、规模等一致；不一致或无法确认时标注风险。
4. 对政策、标准、规划、批复、审查意见等依据资料，应提取文件名称、文号、发布日期、发布机关、适用范围，并判断有效性：现行有效 / 疑似废止 / 已废止 / 已被替代 / 部分条款失效 / 无法判断。
5. 如需要确认政策有效性，可以使用 web_search/web_extract 检索官方发布页、废止公告或替代文件；必须记录真实 URL。找不到依据时写“无法判断，建议人工核实”，不得编造。
6. 输出应重点说明每个文件能否用于本项目分析、可用于哪些 HB 模块、需要人工复核的问题和建议补充资料。

【输出结构】
一、资料验证总判断
请输出：可直接用于分析 / 可用于初判但需补充核实 / 关键资料不足 / 存在无关或失效资料风险。

二、文件逐项验证表
请按“文件名 / 文件类型 / 可读性 / 与本项目相关性 / 可用于模块 / 有效性状态 / 主要依据 / 风险等级 / 处理建议”输出。

三、政策或依据文件有效性识别
请按“文件名称 / 文号 / 发布机关 / 发布日期 / 效力状态 / 识别依据 / 官方URL / 是否建议用于本项目”输出。

四、图片、图纸、扫描件识别摘要
如有图片或扫描件，请说明识别到的关键信息、可用性和识别不确定性。

五、资料缺口和补充建议
请按“缺失资料 / 影响模块 / 影响程度 / 建议提供方 / 优先级”输出。

六、人工复核清单
列出需要工程师人工确认的文件、有效性、项目关联性和关键事实。

【结构化输出】
请在 Markdown 后追加 BEGIN_NODE_RESULT_JSON/END_NODE_RESULT_JSON，JSON 至少包含：
{
  "overall_status": "...",
  "files": [
    {
      "file_name": "...",

      "relevance": "...",

      "validity_status": "...",
      "risk_level": "...",
      "recommendation": "..."
    }
  ],
  "evidence_urls": [{"title": "...", "url": "..."}],
  "manual_review_items": ["..."]
}

【固定免责声明】
输出末尾必须追加：以上为AI辅助初步研判，最终结论需由环评工程师人工复核确认。
